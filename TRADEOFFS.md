# Trade-offs

## Architecture - Why did you structure it this way?

I structured the program using api, application, domain and infrastructure because I found it orderly and easy to access as well as point where a bug may be coming from. For example, if I get a stack trace error coming from PaymentRepository or any of the bank exceptions, I know the issue is from the bank client and I know it definitely comes from the infrastructure folder. I want the api to handle everything related to controllers as it will contain the endpoints used in the program. I want the application to handle the business logic used throughout the application. The infrastructure to handle other concerns as well as database communication with the mockbank. The domain handles entity representation of database as well as fixed states in enums. Basically, I used the structure to put related files together.

## State Management - How do you track payment state? Why?

The payment state is stored in the payments table using a status field that represents the current lifecycle of the payment, such as AUTHORIZED, CAPTURED, VOIDED, or REFUNDED. I introduced a separate payment_operations table to manage idempotency and keep track of individual operations. Separating the two responsibilities keeps the payment record focused on its current state while the operation table records how each request was processed.

## Failure Handling - What's your retry strategy? How do you handle partial failures?

For cases of partial failures like network connectivity issues, the payment status is set to PENDING, until it is retried. The payment is not necessarily validated until it is retried again before it can be authorized. 

## Idempotency - How did you implement it? What edge cases did you consider?

I implemented idempotency by adding a request header called "Idempotency-Key" that is peculiar to each payment request. This header would be generated by the client and passed to the header for each request. Since each idempotency-key header is peculiar to a payment request, it is stored as a unique identifier to each payment in the database. The goal of idempotency in this project is to ensure that even if the same payment request is sent multiple times, the request has the same effect as if it was sent once. 

# Edge cases I considered
- Same key + same request returns the cached result.
- Same key + different request returns 409 Conflict.
- Client retries after timeout do not create duplicate payments.
- Concurrent duplicate requests are partially protected by database constraints but could be improved with stronger concurrency controls.
- Partial failures between the bank and the database would require more robust recovery mechanisms in a production environment.

## What I Would Do Differently - With more time or in production, what would change?

With more time, I would move more payment business logic out of the service class into dedicated domain components. I would also introduce a formal state machine so invalid transitions are prevented in one central place rather than through repeated status checks.

I would not retry every failure. Validation errors and bank declines are permanent and should be returned immediately. Network timeouts and temporary bank outages could be retried with exponential backoff and jitter. I would also add a circuit breaker to stop sending requests when the bank is unavailable.
